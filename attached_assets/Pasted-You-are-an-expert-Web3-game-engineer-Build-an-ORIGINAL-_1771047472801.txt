It must be a production-ready web game where ONLY users who own a verified NFA can play, and the selected NFA becomes the player identity, skin, and progression.

GOAL

- Gating: user must connect wallet and prove NFA ownership (and optionally NFA verification) to enter.
- The run is attributed to the selected NFA tokenId. Leaderboards show NFA name + tokenId + score.

WEB3 + NFA REQUIREMENTS (CRITICAL)
- The user must connect an EVM wallet (MetaMask).
- Query owned NFAs:
  - Use an ERC-721 compatible interface (balanceOf, tokenOfOwnerByIndex OR use an indexer endpoint).
  - Provide BOTH modes:
    1) On-chain enumeration mode (if contract supports ERC721Enumerable)
    2) Indexer mode (preferred): call an HTTP endpoint to fetch NFAs owned by an address
- Verification:
  - Each NFA must be verified by one of:
    A) On-chain: call a registry contract method like isVerifiedNFA(contract, tokenId) -> bool
    B) Off-chain: indexer returns “verified: true”
- In /src/web3, implement a “NFAProvider” with adapters:
  - OnChainEnumerableAdapter
  - IndexerAdapter
  - RegistryVerificationAdapter
  - MockAdapter (for local dev)

LEADERBOARD (SERVER REQUIRED)
- Express + SQLite backend in /server.
- Tables:
  - scores(id, createdAt, chainId, address, nfaContract, tokenId, nfaName, verified, score, distance, durationMs, signature, checksum)
- Endpoints:
  - GET /leaderboard?mode=daily|weekly|alltime&limit=50
  - POST /submit
- Submission security:
  - Client must sign a message with wallet (EIP-191 personal_sign):
    “Honeycomb NFA Tunnel Dash submission:
     chainId: X
     contract: 0x...
     tokenId: 123
     score: 9999
     nonce: <server nonce>”
  - Server issues nonce: GET /nonce?address=0x...
  - Server verifies:
    - signature belongs to address
    - address owns tokenId (via indexer or on-chain call)
    - token is verified if required
    - score sanity checks (distance/time bounds, max pickup rate)
    - checksum format and internal validation
- If verification fails, reject with clear error.
